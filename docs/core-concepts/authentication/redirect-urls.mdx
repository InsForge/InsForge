# Redirect URLs

When using authentication flows that redirect users (OAuth sign-in, email verification links, password reset links), InsForge validates the target URL against a configurable allowlist. This prevents open-redirect attacks where an attacker tricks a user into authenticating and then redirects them to a malicious site.

## How It Works

1. You configure **Allowed Redirect URLs** in the Auth settings page of your InsForge dashboard.
2. When an auth flow includes a `redirectTo` parameter, InsForge checks it against the allowlist.
3. If the URL matches any pattern in the list, the redirect proceeds. Otherwise, the request is rejected.

<Info>
If no redirect URLs are configured (empty list), InsForge allows **all** redirect URLs. This is intentional — it reduces friction during development. Configure your allowlist before going to production.
</Info>

## Supported Patterns

InsForge supports **glob patterns** for redirect URL matching, compatible with [Supabase's redirect URL patterns](https://supabase.com/docs/guides/auth/redirect-urls).

### Pattern Syntax

| Pattern | Matches | Example |
|---------|---------|---------|
| `*` | Any characters except `/` (matches across `.`) | A subdomain or path segment |
| `**` | Any characters including `/` | Multiple path segments |
| `?` | Exactly one character that is not `/` | A single character |
| `[abc]` | Any one character in the set | Character class |
| `[!abc]` | Any one character NOT in the set | Negated character class |

<Note>
`*` matches across `.` boundaries. For example, `https://*.example.com` matches both `https://app.example.com` and `https://deep.sub.example.com`. This is preserved for back-compat with the previous `*.host.com` matcher; if you need to allow only direct subdomains, list each pattern explicitly.
</Note>

### Common Configurations

#### Allow any subdomain

```
https://*.example.com
```

Matches: `https://app.example.com`, `https://staging.example.com`, `https://dev.api.example.com`

Does **not** match: `https://example.com` (the apex domain — add it separately if needed)

#### Allow any path on a domain

```
https://example.com/**
```

Matches: `https://example.com/auth/callback`, `https://example.com/dashboard`, `https://example.com/any/deep/path`

#### Allow a single path segment

```
https://example.com/*
```

Matches: `https://example.com/callback`, `https://example.com/dashboard`

Does **not** match: `https://example.com/auth/callback` (multiple segments — use `**` instead)

#### Combine subdomain and path wildcards

```
https://*.example.com/auth/*
```

Matches: `https://app.example.com/auth/callback`, `https://staging.example.com/auth/verify`

Does **not** match: `https://app.example.com/dashboard`, `https://other.com/auth/callback`

#### Query string wildcards

```
https://example.com/?session=*
```

Matches: `https://example.com/?session=abc123`

#### Multiple exact URLs

```
https://myapp.com
https://myapp.com/auth/callback
https://staging.myapp.com
```

Each URL is matched exactly (no wildcards needed for literal URLs).

## Example: Migrating from Supabase

If you previously used Supabase redirect URL patterns, they work the same way in InsForge:

| Supabase Pattern | InsForge Pattern | Result |
|---|---|---|
| `https://*.example.com` | `https://*.example.com` | ✅ Same behavior |
| `https://example.com/*` | `https://example.com/*` | ✅ Same behavior |
| `https://example.com/**` | `https://example.com/**` | ✅ Same behavior |
| `https://*.example.com/auth/*` | `https://*.example.com/auth/*` | ✅ Same behavior |

## Security Notes

<Warning>
**Be careful with `**` (double-star) patterns.** A pattern like `https://example.com/**` allows redirects to *any* path on that domain, including admin panels, internal API endpoints, and other sensitive routes. Only use `**` when you intentionally want to allow all paths.
</Warning>

### Protocol and Port Are Always Strict

Wildcards never cross protocol or port boundaries:

- `https://*.example.com` does **not** match `http://app.example.com` (protocol mismatch)
- `https://example.com:3000/*` does **not** match `https://example.com:3001/foo` (port mismatch)

### Apex Domain Matching

`*.example.com` does **not** match `example.com` itself (the apex domain). If you need to allow both subdomains and the apex, add two entries:

```
https://*.example.com
https://example.com
```

This is intentional — silently including the apex when only subdomains are configured could be a security footgun.

### No Domain Confusion

`*.example.com` does **not** match `evil-example.com`. The `*` wildcard only matches characters that are part of the subdomain structure of `example.com`.

## Dashboard Configuration

Navigate to **Auth → Settings → General** in your InsForge dashboard to configure allowed redirect URLs. The input accepts both exact URLs and glob patterns described above.
